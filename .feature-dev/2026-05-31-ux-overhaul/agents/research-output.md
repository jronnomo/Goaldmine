## Section 3 — Risks & Conflicts

### RISK-1 — `error.tsx` prop name: `unstable_retry` not `reset` [BLOCKER]

PRD §4.4, CLAUDE.md §4.4, and REQ-A5 all say `{ error, reset }`. **Next 16 uses `unstable_retry`** (confirmed from `node_modules/next/dist/docs/.../10-error-handling.md`). If a developer writes `{ error, reset }` the TypeScript check (`npx tsc --noEmit`) will pass (unrecognized props aren't errors in FC destructuring) but the reset button will silently not work because `reset` will be `undefined`. The acceptance criterion says "reset action" works — it won't without `unstable_retry`. **All four streams must use `unstable_retry` in `error.tsx`.**

### RISK-2 — `LogNutritionForm` does NOT currently show a success confirmation [A1/A4]

The requirements (REQ-A1) extract the "LogNutritionForm feedback pattern" as if it includes a success flash. It does not — the form only resets silently (line 39 `formRef.current?.reset()`). The correct reference pattern for the ~1500ms success flash is `ShareWorkout.tsx:27`: `setCopied(true); setTimeout(() => setCopied(false), 1500)`. REQ-A1's hook must introduce this pattern new; REQ-A4 (optionally refactor `LogNutritionForm`) will also be adding new behavior, not just extracting existing behavior.

### RISK-3 — `ThemeToggle` has no `label` prop [B3]

REQ-B3 says "reuse `ThemeToggle`, optionally with a label." `ThemeToggle` takes zero props. The `MoreSheet` developer must render a layout wrapper row and place `<ThemeToggle />` as the right-side control. Do not add props to `ThemeToggle` — instead, the row in `MoreSheet` provides the "Theme" label text and the component provides the button. This is already the expected approach but worth flagging to avoid a prop-addition attempt.

### RISK-4 — `BottomNav` as sheet host means `latestWeight` must come from a parent [B4]

`BottomNav` is `"use client"`. It cannot call Prisma. The PRD suggests accepting `latestWeight` as optional on `LogLauncher` (default `null`). This is correct — the `LogMeasurementForm` already handles `latestWeight: number | null` (its `defaultValue={latestWeight ?? undefined}` at line 22). If `null` is passed, the weight input simply starts empty. The root `layout.tsx` or `page.tsx` would need to pass `latestWeight` down to `BottomNav` for pre-population. Alternatively, `BottomNav` / `LogLauncher` can skip the prop entirely and leave the weight input empty. Confirm this decision before Stream B starts. **Simplest path: omit `latestWeight` from `BottomNav`; `LogLauncher` passes `null` to `LogMeasurementForm`.**

### RISK-5 — `AppHeader` still renders `ThemeToggle` after `MoreSheet` adds one [B/D]

The header will have two `ThemeToggle` instances: one in `AppHeader` and one in `MoreSheet`. This may be intentional (the header toggle remains) but it is a UX duplication. The PRD does not explicitly say to remove the header toggle. Developers should leave `AppHeader` unchanged unless there is an explicit decision to remove the header toggle. Flag this for the architect to decide.

### RISK-6 — `logNote` missing `revalidatePath("/journal")` [A3]

`logNote` at line 36–52 calls `revalidatePath("/")` and `revalidatePath("/history")` but not `revalidatePath("/journal")`. After Stream A adds the success confirmation, the Journal page will still serve stale data after a note is saved via the Log sheet. Add `revalidatePath("/journal")` to `logNote` as part of REQ-A3.

### RISK-7 — `logMeasurement` missing `revalidatePath("/stats")` and `revalidatePath("/progress")` [A2]

`logMeasurement` (lines 32–33) calls `revalidatePath("/")` and `revalidatePath("/history")` only. After a weigh-in, `/stats` (readiness score uses weight) and the new `/progress` page will be stale. Add both as part of REQ-A2. This is **pre-confirmed** by the PRD §4.3 note.

### RISK-8 — Rest-day detection edge case: `workoutTemplate` present but all blocks are baseline-only [D1]

`page.tsx` already filters `dayBlocks` to exclude blocks where all exercises are baseline test names. If a day's `workoutTemplate` has only baseline blocks, `dayBlocks.length === 0` after filtering but `workoutTemplate !== null`. REQ-D1 says "Rest day when the resolved day has no non-baseline workout blocks (`dayBlocks.length === 0` and no baselines due)" — but `baselinesDue` can be non-empty even on a rest day per se (baseline week schedule overlaps). The safest rest-day condition: `dayBlocks.length === 0 AND workoutTemplate === null`. If `workoutTemplate !== null` but `dayBlocks.length === 0`, it is a baseline-only day (not a rest day). Stream D should clarify this distinction to avoid showing "Rest day" on a baseline test day.

### RISK-9 — `dateKey(now)` must be computed server-side, not in the client island [D3]

PRD §4.5 is correct: `dateKey(now)` uses USER_TZ. If the `TodayCelebration` client island recomputes `dateKey(new Date())` using `@/lib/calendar`, it will call `userParts()` which uses `Intl.DateTimeFormat` with `timeZone: USER_TZ` — this works fine in the browser (USER_TZ is an env var, not available client-side). `USER_TZ = process.env.USER_TZ ?? "America/Denver"` at `calendar.ts:500` — `process.env.USER_TZ` is undefined in the browser. The client-side `dateKey` call would fall back to `"America/Denver"`, which is the correct default, but only by accident. The correct approach is to pass `dateKey` as a prop from the server component to the client island, as the PRD specifies. Do not call `dateKey` inside the client island.

### RISK-10 — `NutritionToday.tsx` imports `LogNutritionForm` directly [D2]

`NutritionToday` at line 1 imports `LogNutritionForm`. When `showLogForm={false}` is passed, the form is not rendered but the import is still resolved. This is fine — the import does not cause a runtime error or hydration issue. Pass `showLogForm={false}` on the Today page after Stream D demotes logging to the Log sheet.

### RISK-11 — `bullseye-pop` keyframe affects `transform` AND `opacity` on the `<svg>` [D3]

The `Bullseye` component renders `<svg>` with no explicit `opacity` or `transform` baseline. Adding `.bullseye-pop` via `className` prop will apply the 320ms animation directly to the SVG. This is correct behavior — but note that if the `DayBullseye` wrapper element (in whatever layout container holds it) has `overflow: hidden`, the `scale(1.08)` at 60% may clip. Stream D should ensure the Bullseye's immediate container allows overflow for the pop moment (`overflow: visible`).
