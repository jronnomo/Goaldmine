### 4.2 `normalizeOffProduct` — Full Rules

**Pipe sanitization at normalizer level.** Strips `|` from `name` and `brand` on ingest, before writing to DB. This gives `LibraryFood.name/brand` a DB-wide pipe-free guarantee — all consumers (chip render, `onAdd` line builder, MCP response) are unconditionally safe without each one having to remember the hazard.

```typescript
export function normalizeOffProduct(raw: OffProduct, barcode: string): {
  name: string;
  brand: string | null;
  servingSize: string | null;
  basis: "serving" | "100g";
  macros: FoodMacros;
} {
  // ── Name ────────────────────────────────────────────────────────────────
  const name = (raw.product_name?.trim().replace(/\|/g, "-") || barcode);

  // ── Brand ────────────────────────────────────────────────────────────────
  const brandRaw = raw.brands?.split(",")[0]?.trim() ?? null;
  const brand = brandRaw ? brandRaw.replace(/\|/g, "-") : null;

  // ── Basis selection ──────────────────────────────────────────────────────
  const hasServingData =
    !!raw.serving_size?.trim() &&
    (
      raw["energy-kcal_serving"] != null ||
      raw["energy_serving"] != null ||
      raw.nutriments?.proteins_serving != null ||
      raw.nutriments?.carbohydrates_serving != null ||
      raw.nutriments?.fat_serving != null
    );

  const basis: "serving" | "100g" = hasServingData ? "serving" : "100g";
  const servingSize = hasServingData ? (raw.serving_size?.trim() ?? null) : "100 g";

  // ── Helpers ──────────────────────────────────────────────────────────────
  const suffix = basis === "serving" ? "_serving" : "_100g";

  function safeNum(v: number | undefined | null): number | null {
    if (v == null || !Number.isFinite(v) || v < 0) return null;
    return v;
  }

  // ── Calories ────────────────────────────────────────────────────────────
  // Priority 1: energy-kcal_{suffix} (direct kcal field)
  // Priority 2: energy_{suffix} in kJ ÷ 4.184
  // NEVER derive calories from macros × Atwater. NEVER use bare "energy" as kcal.
  let calories: number | null = null;
  const kcalDirect = safeNum(
    basis === "serving" ? raw["energy-kcal_serving"] : raw["energy-kcal_100g"]
  );
  if (kcalDirect != null) {
    calories = Math.round(kcalDirect);
  } else {
    const kjVal = safeNum(
      basis === "serving" ? raw["energy_serving"] : raw["energy_100g"]
    );
    if (kjVal != null) {
      calories = Math.round(kjVal / 4.184);
    }
  }

  // ── Protein, Carbs, Fat, Fiber ───────────────────────────────────────────
  function gram1dp(v: number | undefined | null): number | null {
    const n = safeNum(v);
    if (n == null) return null;
    return Math.round(n * 10) / 10;
  }
  const n = raw.nutriments ?? {};
  const proteinG  = gram1dp(n[`proteins${suffix}`]);
  const carbsG    = gram1dp(n[`carbohydrates${suffix}`]);
  const fatG      = gram1dp(n[`fat${suffix}`]);
  const fiberG    = gram1dp(n[`fiber${suffix}`]);

  // ── Sodium ───────────────────────────────────────────────────────────────
  // OFF stores sodium in grams → multiply by 1000 for mg.
  // If sodium missing but salt present: salt_g × 400 (NaCl 39.3% sodium ≈ 400 mg/g).
  let sodiumMg: number | null = null;
  const sodiumRaw = safeNum(n[`sodium${suffix}`]);
  if (sodiumRaw != null) {
    sodiumMg = Math.round(sodiumRaw * 1000);
  } else {
    const saltRaw = safeNum(n[`salt${suffix}`]);
    if (saltRaw != null) {
      sodiumMg = Math.round(saltRaw * 400);
    }
  }

  return { name, brand, servingSize, basis, macros: { calories, proteinG, carbsG, fatG, fiberG, sodiumMg } };
}
```

### 5.1 `lookupBarcode(raw: string): Promise<BarcodeLookupResult>`

**Multi-form barcode match set** — EAN-8 / UPC-A / EAN-13 interop:

```typescript
export async function lookupBarcode(raw: string): Promise<BarcodeLookupResult> {
  const barcode = raw.trim();

  // Validate format before any DB/network work
  if (!/^\d{8,14}$/.test(barcode)) {
    return { status: "not_found" };
  }

  // ── Multi-form match set ────────────────────────────────────────────────
  // Rules (binding per Tech Lead):
  //   • EAN-8 (8 digits): looked up as-is — DO NOT zero-pad to 13
  //   • UPC-A (12 digits): padded form = "0" + barcode (EAN-13 equivalent)
  //   • EAN-13 starting with 0: stripped form = barcode.slice(1) (UPC-A equivalent)
  //   • Any other length: only raw form
  const padded13 = barcode.length === 12 ? "0" + barcode : null;
  const stripped = barcode.length === 13 && barcode.startsWith("0") ? barcode.slice(1) : null;
  const forms = [barcode, padded13, stripped].filter((f): f is string => f !== null);

  // Canonical storage key used for upserts — EAN-13 for UPC-A input; raw otherwise
  const canonicalKey = padded13 ?? barcode;

  // ── Library lookup ───────────────────────────────────────────────────────
  const existing = await prisma.foodLibrary.findFirst({
    where: { barcode: { in: forms } },
  });

  if (existing) {
    await prisma.foodLibrary.update({
      where: { id: existing.id },
      data: { usageCount: { increment: 1 }, lastUsedAt: new Date() },
    });
    return { status: "found", food: toLibraryFood(existing), fromLibrary: true };
  }

  // ── OFF fetch (only on library miss) ────────────────────────────────────
  // 12-digit UPC-A miss → retry once with "0" + barcode (EAN-13 form)
  let result = await fetchOff(barcode, canonicalKey);
  if (result.status === "not_found" && barcode.length === 12) {
    result = await fetchOff("0" + barcode, "0" + barcode);
  }
  return result;
}
```

**OFF fetch with exact response discipline**:

```typescript
const OFF_UA = "Goaldmine/1.0 (github.com/jronnomo/goaldmine)";
const OFF_FIELDS = [
  "product_name", "brands", "serving_size",
  "energy-kcal_serving", "energy_serving",
  "energy-kcal_100g", "energy_100g",
  "nutriments",
].join(",");

/**
 * OFF HTTP response discipline (binding):
 *   HTTP 200 + json.status === 1 + json.product present → found
 *   HTTP 200 + json.status === 0                         → not_found (product in OFF but no usable data)
 *   HTTP 404                                             → not_found (product genuinely absent)
 *   Any other non-2xx (429, 500, etc.)                  → error (retryable — OFF may be down)
 *   Network failure / timeout (AbortError)              → error (retryable)
 *
 * Only "not_found" writes nothing to the DB. "error" shows a Retry button.
 * Do NOT map 429/500 to not_found — the product may exist; the request failed.
 */
async function fetchOff(code: string, canonicalKey: string): Promise<BarcodeLookupResult> {
  const url = `https://world.openfoodfacts.org/api/v2/product/${code}?fields=${OFF_FIELDS}`;
  try {
    const res = await fetch(url, {
      headers: { "User-Agent": OFF_UA },
      signal: AbortSignal.timeout(6000),
    });

    if (!res.ok) {
      // 404 = product not in OFF. Any other non-2xx = transient failure (rate limit, server error).
      return res.status === 404
        ? { status: "not_found" }
        : { status: "error", message: "OpenFoodFacts unavailable — try again" };
    }

    const json = await res.json() as { status: number; product?: unknown };
    // 200 + status:1 = found. 200 + status:0 = not_found (incomplete OFF record).
    if (json.status !== 1 || !json.product) return { status: "not_found" };

    const normalized = normalizeOffProduct(json.product as OffProduct, code);

    const upserted = await prisma.foodLibrary.upsert({
      where: { barcode: canonicalKey },
      create: {
        barcode: canonicalKey,
        name: normalized.name,
        brand: normalized.brand,
        servingSize: normalized.servingSize,
        basis: normalized.basis,
        ...normalized.macros,
        usageCount: 1,
        lastUsedAt: new Date(),
      },
      update: {
        // Re-normalize on re-encounter: OFF data can improve over time
        name: normalized.name,
        brand: normalized.brand,
        servingSize: normalized.servingSize,
        basis: normalized.basis,
        ...normalized.macros,
        usageCount: { increment: 1 },
        lastUsedAt: new Date(),
      },
    });

    // revalidatePath("/nutrition") is safe while the Log form is open:
    // It invalidates the RSC router cache so the server component re-renders with the
    // new quickPickFoods prop. React does NOT destroy client component state on RSC
    // re-renders — client component instances persist across RSC payload refreshes
    // unless their DOM position changes. The Log form's controlled state (itemsText,
    // macros, mealType) survives intact. See integration smoke item §11.5.
    // If form field clobbering is observed in practice, move this call to recordFoodUse
    // (post-submit only) and remove it from here.
    revalidatePath("/nutrition");

    return { status: "found", food: toLibraryFood(upserted), fromLibrary: false };
  } catch {
    return { status: "error", message: "Network error — try again" };
  }
}
```

### 6.3 Camera Lifecycle — Start-Generation Counter (H-1 Fix)

**Root cause of the track leak**: `getUserMedia()` is async. If `stopCamera()` runs while `getUserMedia()` is in-flight, `streamRef.current` is cleared. When `getUserMedia()` later resolves, `startCamera()` continues and sets `streamRef.current = stream` — a stream whose tracks are now running but which `stopCamera()` never had a reference to stop. On iOS, the green camera-in-use indicator stays on after the sheet closes.

**Fix — start-generation counter**: `startGenRef` is a monotonically increasing integer. `stopCamera()` increments it to invalidate the current generation. When `getUserMedia()` resolves, `startCamera()` checks whether its captured generation still matches the current counter. If not, it immediately stops the fresh tracks and returns. This covers every race: close during starting, visibilitychange restart, double-startCamera, etc.

**Full pseudocode**:

```typescript
// Lifecycle refs
const startGenRef = useRef(0);     // H-1 cancellation counter
const streamRef   = useRef<MediaStream | null>(null);
const decoderRef  = useRef<Decoder | null>(null);

async function startCamera() {
  const gen = ++startGenRef.current; // capture THIS call's generation BEFORE the await
  setScannerState("starting");
  try {
    const stream = await navigator.mediaDevices.getUserMedia({
      video: { facingMode: { ideal: "environment" }, width: { ideal: 1280 }, height: { ideal: 720 } },
    });
    // Guard: if stopCamera() ran (or another startCamera()) while we were awaiting, abort
    if (gen !== startGenRef.current) {
      stream.getTracks().forEach(t => t.stop()); // immediately release the fresh tracks
      return;
    }
    streamRef.current = stream;
    if (videoRef.current) {
      videoRef.current.srcObject = stream;
      await videoRef.current.play();
    }
    // Guard again after play() — play() can also take non-trivial time
    if (gen !== startGenRef.current) {
      stream.getTracks().forEach(t => t.stop());
      streamRef.current = null;
      return;
    }
    await initDecoder();
    if (gen !== startGenRef.current) return; // superseded during wasm warm-up
    setScannerState("scanning");
  } catch (err) {
    if (gen !== startGenRef.current) return; // already superseded — ignore error
    const name = (err as DOMException).name;
    if (name === "NotAllowedError" || name === "PermissionDeniedError") {
      setScannerState("denied");
    } else {
      setScannerState("no-camera");
    }
  }
}

function stopCamera() {
  startGenRef.current++; // invalidate any in-flight startCamera — MUST increment BEFORE track.stop()
  streamRef.current?.getTracks().forEach(t => t.stop());
  streamRef.current = null;
  if (videoRef.current) videoRef.current.srcObject = null;
  decoderRef.current = null;
  ctxRef.current = null; // clear cached canvas context
}
```

### 6.4 Decoder Strategy — Initialization (once per camera session)

```typescript
type Decoder = (imageData: ImageData) => Promise<string | null>;
const decoderRef = useRef<Decoder | null>(null);

async function initDecoder() {
  // Native BarcodeDetector path (Chrome Android, Chrome desktop)
  // Format names use underscore in the native API: "ean_13", NOT "EAN-13"
  if (typeof BarcodeDetector !== "undefined") {
    try {
      const supported = await BarcodeDetector.getSupportedFormats();
      if (supported.includes("ean_13")) {
        const detector = new BarcodeDetector({
          formats: ["ean_13", "upc_a", "upc_e", "ean_8"],
        });
        decoderRef.current = async (img) => {
          const results = await detector.detect(img);
          return results[0]?.rawValue ?? null;
        };
        return;
      }
    } catch {
      // getSupportedFormats failed — fall through to zxing
    }
  }

  // zxing-wasm path (iOS Safari, Firefox, everywhere native is absent)
  const { prepareZXingModule, readBarcodes } = await import("zxing-wasm/reader");
  await prepareZXingModule({
    overrides: {
      locateFile: (path: string, prefix: string) => {
        if (path.endsWith(".wasm")) return `/zxing/${path}`;
        return prefix + path;
      },
    },
    fireImmediately: true,
  });

  decoderRef.current = async (img) => {
    const results = await readBarcodes(img, {
      formats: ["EAN-13", "UPC-A", "UPC-E", "EAN-8"], // HRI labels valid in v3.1.0
      tryHarder: true,
      maxNumberOfSymbols: 1,
    });
    const valid = results.find(r => r.isValid && r.text);
    return valid?.text ?? null;
  };
}
```

### 6.5 Decode Loop (runs while `scannerState === "scanning"`)

**Canvas context** (`willReadFrequently: true`): Initializing with this flag tells the browser to keep the canvas in CPU-readable memory, avoiding a GPU→CPU readback on every `getImageData()`. On a 200 ms decode loop without this flag, each readback can spike 2–5 ms on mobile. The context is cached via `ctxRef` and reused across all frames in a session. `stopCamera()` clears `ctxRef.current` so the next session creates a fresh context on the same canvas element (`getContext` is idempotent and returns the same underlying context if called again).

```typescript
const ctxRef = useRef<CanvasRenderingContext2D | null>(null);

useEffect(() => {
  if (scannerState !== "scanning" || !decoderRef.current) return;

  let cancelled = false;
  let inFlight = false;
  let lastCode: string | null = null;
  let matchCount = 0;

  const loop = async () => {
    while (!cancelled) {
      await new Promise<void>(r => setTimeout(r, 200));
      if (cancelled || inFlight) continue;

      const video = videoRef.current;
      // S-3 fix: null guards instead of non-null assertions (loop can run after unmount)
      const canvas = canvasRef.current;
      if (!video || !canvas || video.readyState < HTMLMediaElement.HAVE_CURRENT_DATA) continue;

      inFlight = true;
      try {
        // M-4 fix: initialize ctx once per session with willReadFrequently
        if (!ctxRef.current) {
          ctxRef.current = canvas.getContext("2d", { willReadFrequently: true });
        }
        const ctx = ctxRef.current;
        if (!ctx || !decoderRef.current) { inFlight = false; continue; }

        const W = Math.min(video.videoWidth, 1280);
        const H = Math.round(W * (video.videoHeight / video.videoWidth));
        canvas.width = W;
        canvas.height = H;
        ctx.drawImage(video, 0, 0, W, H);
        const imageData = ctx.getImageData(0, 0, W, H);

        const code = await decoderRef.current(imageData);
        if (!cancelled && code) {
          if (code === lastCode) {
            matchCount++;
            if (matchCount >= 2) {
              navigator.vibrate?.(50);
              onDetected(code);
              return;
            }
          } else {
            lastCode = code;
            matchCount = 1;
          }
        }
      } catch {
        // decode errors are transient; continue loop
      } finally {
        inFlight = false;
      }
    }
  };

  loop();
  return () => { cancelled = true; };
}, [scannerState, onDetected]);
```

### 7.2 Stacked-Dialog Behavior Contract

**What "stacked" means here**: `LogNutritionForm` renders `ScanFoodSheet` as a sibling of (or outside) the Log `BottomSheet`. Both use the same `BottomSheet` component which calls `dialog.showModal()`. When the user taps "Scan," ScanFoodSheet's dialog calls `showModal()` while the Log dialog is already modal.

**Native `<dialog>` top-layer semantics (binding)**:
- The second `showModal()` pushes ScanFoodSheet's dialog above the Log dialog in the top layer.
- `Esc` fires `cancel` on the topmost dialog only — ScanFoodSheet closes; Log dialog stays open.
- Backdrop click on ScanFoodSheet's `::backdrop` closes ScanFoodSheet only; Log dialog and its form state are unaffected.
- Body `overflow:hidden` scroll-lock is safe: the inner dialog restores to its prior value on close; the outer dialog's lock is still in effect.
- Focus is trapped to the topmost dialog while it is open; on ScanFoodSheet close, focus returns to the element that opened it (the Scan button in the form).

**Verified status**: The BadgeWall pattern (previously cited as a "confirmed" precedent) is NOT a stacked-dialog precedent — BadgeWall opens a BottomSheet from a page, not from inside another BottomSheet. No instance of nested `showModal()` has been tested in this codebase. The behavior contract above is theoretically sound per the HTML spec but requires an integration smoke test before shipping (see §11.5).

**BottomSheet cleanup**: If `ScanFoodSheet`'s `<dialog>` is open and `LogNutritionForm` somehow unmounts, the native `dialog` stays in the DOM as an orphan with no React tree. In practice the focus trap prevents this path. As a defensive measure, confirm `BottomSheet.tsx`'s show-effect returns a cleanup function: `return () => { if (dialogRef.current?.open) dialogRef.current.close(); }`. If this return is missing, add it as part of Stream B.

**Fallback**: If `showModal()`-over-`showModal()` misbehaves on iOS in practice (rendering artifacts, Esc closing wrong dialog, focus trap failure), the recovery path is: close the Log sheet while the Scan sheet is open (one sheet visible at a time). This is explicitly NOT preferred for UX; use it only on observed, unreproducible-in-spec breakage.

### 8.3 `handleAdd` — Single Canonical Implementation

**There is exactly one version of this function.** The `chipSource: boolean` field in `AddFoodPayload` distinguishes who calls `recordFoodUse`. The rule is:

- `chipSource === true`: chip tap path. `lookupBarcode` was NOT called. `handleAdd` calls `recordFoodUse` fire-and-forget.
- `chipSource === false`: scan path. `lookupBarcode` already bumped `usageCount` + `lastUsedAt`. `handleAdd` must NOT call `recordFoodUse` (would double-count).

```typescript
function handleAdd({ food, servings, chipSource }: AddFoodPayload) {
  // ── 1. Build items line ──────────────────────────────────────────────────
  // name and brand are pipe-free (normalizer-level DB guarantee — no replace needed)
  const brandPart = food.brand ? ` (${food.brand})` : "";
  const qty = food.basis === "100g"
    ? `${Math.round(servings * 100)} g`
    : `${servings} serving${servings === 1 ? "" : "s"}`;
  const line = `${food.name}${brandPart} | ${qty}`;
  setItemsText(prev => prev + (prev.trim() ? "\n" : "") + line);

  // ── 2. Sum macros ────────────────────────────────────────────────────────
  // null food values are skipped — they do not zero-out an existing manual entry.
  // cal/sodium: integer. gram fields: 1 decimal.
  setMacros(prev => {
    const next = { ...prev };
    for (const key of MACRO_KEYS) {
      const foodVal = food.perServing[key];
      if (foodVal == null) continue;
      const scaled = foodVal * servings;
      const existing = prev[key] ?? 0;
      const sum = existing + scaled;
      if (key === "calories" || key === "sodiumMg") {
        next[key] = Math.round(sum);
      } else {
        next[key] = Math.round(sum * 10) / 10;
      }
    }
    return next;
  });

  // ── 3. Usage bump (chip path only) ───────────────────────────────────────
  // Scan path: lookupBarcode() already incremented usageCount. Do NOT call here.
  // Chip path: lookupBarcode() was NOT called. Bump here fire-and-forget.
  if (chipSource) {
    recordFoodUse(food.id).catch(() => {});
  }

  // ── 4. Optimistic chip prepend (scan path only) ──────────────────────────
  // After a scan adds a food that is not yet in the visible chips,
  // prepend it immediately — no refetch round-trip needed.
  // Cap at 8. If already in the list (library hit already visible), skip.
  if (!chipSource) {
    setQuickPick(prev => {
      if (prev.some(f => f.id === food.id)) return prev;
      return [food, ...prev].slice(0, 8);
    });
  }
}
```

### 8.7 revalidatePath While Form Open — Verified Behavior

`lookupBarcode()` calls `revalidatePath("/nutrition")` immediately after a successful OFF upsert. This invalidates the Next.js RSC router cache. The server component (`NutritionPage`) re-renders and sends a fresh RSC payload to the client, which includes an updated `quickPickFoods` array.

**Client component state is NOT destroyed**: React does not unmount and remount client component instances on RSC payload re-renders. `LogNutritionForm`'s `useState` values (`itemsText`, `macros`, `mealType`, `quickPick`, `scanOpen`) all survive intact. The `quickPickFoods` sync effect (§8.2) accepts the new prop array and updates `quickPick` state to reflect the server's authoritative ordering.

**Integration smoke item** (see §11.5): scan-add a new food with the items textarea partially filled → confirm the textarea text persists post-add. If field clobbering is observed (text reset, macros zeroed), move `revalidatePath` out of `fetchOff` and into `recordFoodUse` / post-submit only, so it only fires after the user's session ends.

## 9. MCP Change — `get_nutrition_history` `frequentFoods` Slot-In

**Location**: `src/lib/mcp/tools.ts`, inside the `get_nutrition_history` handler, after the `byDay` computation.

**frequentFoods uses an INDEPENDENT try/catch** — it must not abort the tool if the FoodLibrary table is unavailable (migration not yet applied, query error, etc.):

```typescript
// frequentFoods — top-5 by usageCount for coach macro estimates
// Independent try/catch: FoodLibrary failure degrades to [] without breaking nutrition history.
// The blueprint states "If the table doesn't exist, the build will fail" — this is INCORRECT.
// The Prisma client is generated from the schema regardless; the table absence is a RUNTIME
// failure. An independent try/catch ensures get_nutrition_history continues serving byDay data
// even before the migration is applied.
let frequentFoods: Array<{
  name: string; brand: string | null; servingSize: string | null; basis: string;
  perServing: { calories: number | null; proteinG: number | null; carbsG: number | null;
    fatG: number | null; fiberG: number | null; sodiumMg: number | null };
}> = [];

try {
  const frequentFoodRows = await prisma.foodLibrary.findMany({
    orderBy: [{ usageCount: "desc" }, { lastUsedAt: "desc" }],
    take: 5,
    select: {
      name: true, brand: true, servingSize: true, basis: true,
      calories: true, proteinG: true, carbsG: true, fatG: true,
      fiberG: true, sodiumMg: true,
    },
  });
  frequentFoods = frequentFoodRows.map(f => ({
    name: f.name,
    brand: f.brand,
    servingSize: f.servingSize,
    basis: f.basis,
    perServing: {
      calories: f.calories, proteinG: f.proteinG, carbsG: f.carbsG,
      fatG: f.fatG, fiberG: f.fiberG, sodiumMg: f.sodiumMg,
    },
  }));
} catch {
  // FoodLibrary migration not yet applied, or query error — return empty array.
  // byDay data is unaffected.
}

return {
  since,
  days,
  mealType: mealType ?? null,
  entryCount: rows.length,
  frequentFoods, // always present; [] when library is empty or unavailable
  byDay,
};
```

**Notes**:
- `frequentFoods: []` is always present in the response (not a missing key) — both when the library is empty and when the query fails.
- No new tool registered. Tool description unchanged — `frequentFoods` is an additive field.
- Curl verification: `get_nutrition_history` response must include `"frequentFoods": []` before migration is applied (not an error, not a missing key).

## 13. Critical Decisions with Reasoning

| # | Decision | Reasoning | Alternative Considered |
|---|----------|-----------|----------------------|
| 1 | **Pipe sanitization at normalizer level** | `LibraryFood.name/brand` carry a DB-wide pipe-free guarantee; all consumers are unconditionally safe. Single ingest-time strip vs. remembering the hazard at every consumer. | Sanitize in `onAdd` — rejected; new consumers inherit the raw-pipe risk silently. |
| 2 | **`zxing-wasm` exact pin `"3.1.0"`** | API researched against this exact version. Exact pin = zero surprise on `npm ci`. | `^3.1.0` — rejected; wasm packages can have breaking minor bumps. |
| 3 | **Decoder initialized once per camera session** | `prepareZXingModule` warms the wasm; `getSupportedFormats` is async; both stable for a session. Per-frame init is a performance regression. | Per-frame constructor — rejected. |
| 4 | **Two consecutive identical reads** | Prevents spurious reads from partial scans and motion blur. Two reads at 200ms loop = ≤600ms total — well within perceived real-time. | Single read — false positives. Three reads — too slow (>800ms). |
| 5 | **Manual strip always rendered, no camera state blocks it** | US-004 ("never blocked"). Strip lives in ScanFoodSheet, not inside BarcodeScanner. Camera states only affect the viewfinder. | Hide strip during scanning — rejected; breaks fallback UX. |
| 6 | **`initialFood` bypasses scan phase entirely** | Chip taps feel instant; no camera permission fires; no lifecycle cost for the common chip case. | Always open scan phase — rejected; unnecessary camera churn. |
| 7 | **LogNutritionForm dual-fetch strategy** | `/nutrition` page is server component (zero extra latency). LogLauncher is client-only; mount = accordion open fires lazy fetch. No root-layout DB threading required. | Fetch in LogLauncher — rejected; would need server wrapper. |
| 8 | **frequentFoods in independent try/catch** | Migration may not be applied at deploy time. The tool must continue serving `byDay` data regardless. Atomic `safe()` would break ALL nutrition history reads on a runtime FoodLibrary error. | Same `safe()` block — rejected; partial failure kills full tool response. |
| 9 | **Canonical barcode storage key = padded EAN-13 for UPC-A; raw otherwise** | EAN-13 is the universal standard. Storing padded-13 for 12-digit UPC-A means the `@unique` index is always hit consistently. EAN-8 (8 digits) stored as-is per PRD §3.1.3 — zero-padding EAN-8 to 13 produces a meaningless code that is NOT the EAN-13 for the same product. | Pad all barcodes to 13 — rejected; EAN-8 padding violates the barcode specification. |
| 10 | **`chipSource: boolean` (required, not optional)** | An optional field with a default of `false` is a subtle trap — callers that forget to pass it would silently choose the wrong branch. A required `boolean` forces every call site to make the decision explicit. | `chipSource?: boolean` — rejected; optional booleans default silently. |
| 11 | **Start-generation counter (H-1 fix)** | A simple monotonically increasing integer captures every close path (Esc, backdrop, route change, visibilitychange) without requiring a separate AbortController per `getUserMedia`. The counter is checked at every async suspension point in `startCamera`. | Per-call AbortController — more complex; doesn't handle the case where `getUserMedia` itself is not abortable on all browsers. |
| 12 | **willReadFrequently + ctxRef** | `getImageData` on a GPU-accelerated canvas requires a GPU→CPU readback per call — 2–5 ms on mobile at 200 ms intervals. `{ willReadFrequently: true }` on the first `getContext` call opts the canvas into CPU memory. `ctxRef` caches the context across frames to avoid repeated lookups; cleared in `stopCamera` for a clean per-session reset. | Per-frame `getContext` call — already idempotent (returns same object), but the flag must be on the FIRST call; passing it on a subsequent call after an uncached first call has no effect. |
| 13 | **OFF response discipline: 404 → not_found, other non-2xx → error** | HTTP 429 and 500 are transient failures, not product absences. Mapping them to `not_found` sends the user to the manual path with no retry affordance, permanently suppressing recovery. Only HTTP 404 and `json.status === 0` reliably indicate the product is absent from OFF. | `!res.ok → not_found` (v1) — rejected; misclassifies rate-limit and server errors. |
| 14 | **Optimistic chip prepend in handleAdd** | After a scan add, the food should appear in chips immediately for the common multi-item scan session. Waiting for the server round-trip (revalidatePath → RSC re-render → prop sync) introduces a visible latency. The dedup guard prevents duplicates. The sync effect will later replace the local ordering with the server's authoritative one. | No optimistic update — rejected; chips appear stale during multi-item scan sessions. |
